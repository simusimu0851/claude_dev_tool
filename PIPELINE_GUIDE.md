# AI 개발 파이프라인 v4 — 워크플로우 & 개선사항 가이드

## 1. 핵심 컨셉

이 파이프라인의 목표는 **Claude API를 장기 개발 세션에서 최소 비용으로 사용하는 것**이다.

핵심 전략: **변하지 않는 정보는 캐시, 변하는 정보는 압축, 소비된 정보는 축약**.

```
┌─────────────────────────────────────────────────────────────┐
│                    API 호출 1회 구조                         │
├─────────────────────────────────────────────────────────────┤
│  system prompt          ← 항상 동일, 매 호출 캐시 히트       │
│  ─────────────────────── cache breakpoint 1 ──────────────  │
│  PROJECT.md             ← 세션 간 불변, 크로스세션 캐시      │
│  ─────────────────────── cache breakpoint 2 ──────────────  │
│  SESSION_STATE.md       ← 세션 내 불변, 세션 내 캐시         │
│  ─────────────────────── cache breakpoint 3 ──────────────  │
│  Summary Memory         ← 압축 후 ~40메시지 불변, 캐시       │
│  ─────────────────────── (no breakpoint) ─────────────────  │
│  Sliding Memory         ← 매 메시지 변경, 캐시 불가 (정상)   │
└─────────────────────────────────────────────────────────────┘

캐시 히트 시 해당 구간 토큰 비용 = 원가의 10%
```

## 2. 사용 워크플로우

### 2.1 세션 시작

```bash
python3 dev_pipeline_v4.py
# 또는 설정 커스텀:
python3 dev_pipeline_v4.py --engine unity --path /path/to/project
```

시작 시 자동으로:
1. `pipeline_config.json` 로드 (있으면)
2. `dev_output/PROJECT.md` 로드 → 프로젝트 캐시 블록 구성
3. `dev_output/SESSION_STATE.md` 로드 → 이전 작업 맥락 복원

### 2.2 일반 개발 루프

```
You > PlayerController에 이동 기능 구현해줘
🤖 (스트리밍 응답 + diff 블록)
🔧 diff 블록 감지 — 파일에 적용 중...
  ✅ 패치 적용됨: Assets/Scripts/PlayerController.cs  (백업: .cs.bak)
  [캐시] 히트 3,420t (72%) | 생성 0t

You > 점프도 추가해줘
🤖 (이전 대화 참조하여 일관된 코드 생성)
  [캐시] 히트 4,100t (68%)

You > read Assets/Scripts/PlayerController.cs Update
📄 PlayerController.cs (42줄, 관련 섹션만 발췌) → 컨텍스트 주입

You > 여기서 Ground Check 로직이 이상한데 고쳐줘
🤖 (주입된 파일 컨텍스트 참조하여 수정)
```

**핵심 포인트:**
- 일반 텍스트 입력 → AI에게 개발 요청으로 전달
- 코드 수정 응답은 자동으로 diff 파싱 → 파일 패치 (백업 포함)
- `read` 명령으로 필요한 파일만 선택적으로 컨텍스트에 주입
- 캐시 히트율이 실시간으로 표시됨

### 2.3 명령어 레퍼런스

| 명령어 | 동작 | 토큰 영향 |
|--------|------|----------|
| `session` / `세션종료` | SESSION_STATE.md 저장 + 메모리 초기화 + 재시작 | 세션 상태 생성 ~800t, 이후 캐시 리셋 |
| `save` / `저장` | 중간 저장 (세션 계속) | 세션 상태 생성 ~800t |
| `exit` / `프로그램종료` | 프로그램 종료 (미저장 시 경고) | 0 |
| `project update` | PROJECT.md 재생성 | ~1200t 출력 + **다음 세션 캐시 미스 유발** |
| `read <파일> [검색어]` | 파일(또는 관련 섹션) 컨텍스트 주입 | 파일 크기만큼 (이전 파일 자동 축약) |
| `status` / `상태` | 세션 통계 + 캐시 절약량 | 0 |
| `help` / `도움말` | 도움말 | 0 |

### 2.4 세션 전환 흐름

```
[세션 1: PlayerController 작업]
You > (10~30개 요청으로 기능 구현)
You > session
  📝 세션 상태 저장 중... 완료 (1,240 토큰)
  ✅ SESSION_STATE.md 저장

  📊 토큰 사용량
    cheap        :     45,200
    advisor      :          0
    합계         :     45,200
    캐시 절약    :     28,000  (~38% 절감)

  🔄 새 세션 시작
     PROJECT.md  → 캐시 재사용 가능 (내용 불변)
     SESSION_STATE.md → 새 캐시 생성 (세션 내 유효)

[세션 2: EnemyAI 작업]
  📋 PROJECT.md 로드됨 (1,840자)           ← 캐시 히트 (세션 1과 동일 내용)
  📂 SESSION_STATE.md 로드됨 (620자) → 이전 작업 복원

You > 적 AI 만들어줘
  (PlayerController 작업 맥락을 알고 있으므로 충돌 없이 구현)
```

### 2.5 컨텍스트 압축 (자동)

```
[메시지 50개 초과 또는 총 120,000자 초과 시 자동 발동]
[💭 컨텍스트 압축 중 — 잠시 기다려주세요...]

내부 처리:
  1. 최근 10개 메시지는 보호 (Working Memory)
  2. 나머지 40+개를 LLM으로 요약 (haiku 모델, 저비용)
  3. 요약이 Summary Memory로 저장 → 이후 캐시 가능
  4. Sliding Memory = 최근 10개만 남음
```

## 3. 토큰 최적화 아키텍처

### 3.1 5계층 메모리 시스템

```
계층           크기         변경 빈도      캐시 여부    비고
────────────────────────────────────────────────────────────
Project        ~1200t       거의 없음      ✅ 최강      세션 간 유지
Session        ~800t        세션마다       ✅           세션 내 2번째 호출부터
Summary        가변          압축 시       ✅           압축 후 ~40메시지 유지
Sliding        ~10-50msg    매 메시지      ❌           현재 대화
Working        10msg        매 메시지      ❌           Sliding의 최근 10개 (보호)
```

### 3.2 캐시 비용 구조 (Anthropic 기준)

```
일반 입력 토큰:    $3.00 / 1M tokens (Haiku 기준)
캐시 생성:         $3.75 / 1M tokens (25% 추가)
캐시 히트:         $0.30 / 1M tokens (90% 할인)

→ 2번만 히트해도 캐시 생성 비용 회수
→ 10회 호출 세션에서 PROJECT.md(1200t)가 캐시되면:
   일반: 1200 × 10 = 12,000t 과금
   캐시: 1200 × 1(생성) + 1200 × 9 × 0.1(히트) = 2,280t 상당
   절감: ~81%
```

### 3.3 코드 수정 방식: diff vs 전체 파일

```
전체 파일 (500줄 C# 파일):
  출력: ~2,000 토큰
  
diff (10줄 변경):
  출력: ~100 토큰
  절감: 95%

→ 시스템 프롬프트에서 diff 형식 강제
→ 자동 파싱 + 파일 패치 (백업 포함)
```

## 4. 적용된 개선사항 (v4 최적화)

### 4.1 캐시 효율 개선

| # | 개선 | 위치 | 효과 |
|---|------|------|------|
| 1 | **Summary Memory에 cache_control 추가** | `memory_manager.py` | 압축 후 ~40회 API 호출에서 summary 토큰 90% 할인 |
| 2 | **내부 호출에서 메인 시스템프롬프트 재사용** | `pipeline_controller.py` | SESSION_STATE/PROJECT 생성, 압축 시 시스템프롬프트 캐시 히트 |
| 3 | **compress()에서 시스템프롬프트 필수 전달** | `memory_manager.py` | throwaway 프롬프트 제거 → 캐시 miss 0% → hit 가능 |

### 4.2 토큰 절약 개선

| # | 개선 | 위치 | 효과 |
|---|------|------|------|
| 4 | **파일 컨텍스트 자동 축약** | `memory_manager.py` | `read`로 주입된 이전 파일을 헤더만 남김. 유니티 C# 파일 500~2000줄이 매 API 호출에 포함되던 것 방지 |
| 5 | **메시지 크기 기반 조기 압축** | `memory_manager.py` | 50개 미만이어도 총 120,000자 초과 시 압축 트리거. 코드가 많은 대화에서 토큰 폭증 방지 |
| 6 | **세션 상태 생성에 complete() 사용** | `pipeline_controller.py` | 내부 처리를 stream() → complete()로 교체. 스트리밍 오버헤드 제거 |
| 7 | **섹션 추출 임계값 하향 (300줄→150줄)** | `file_manager.py` | 유니티 파일에서 `read <파일> <검색어>` 시 전문 대신 관련 섹션만 주입 |
| 8 | **압축 모델 haiku 명시** | `memory_manager.py` | config의 cheap_model 변경과 무관하게 압축은 항상 최저비용 |

### 4.3 안정성 개선

| # | 개선 | 위치 | 효과 |
|---|------|------|------|
| 9 | **diff 적용 전 .bak 백업** | `file_manager.py` | 패치 전 원본 복사, 실패 시 자동 복원. 토큰 비용 0 |
| 10 | **캐시 히트 실시간 로깅** | `llm_client.py` | `[캐시] 히트 3,420t (72%)` 형태로 매 호출 표시. 캐시 전략 검증 가능 |

### 4.4 GPT 제안 평가 (project_core/dynamic 분리)

**채택하지 않음.** 이유:
- PROJECT.md는 이미 불변 설계 — 분리할 "dynamic" 부분이 없음
- dynamic 정보는 SESSION_STATE.md에 이미 존재
- 분리하면 cache breakpoint 증가 (3→4개) → fake ack 메시지 추가 비용
- 복잡도만 올라가고 캐시 효율은 동일하거나 악화

## 5. 유니티 게임개발 권장 워크플로우

### 5.1 세션 단위 = 기능 단위

```
세션 1: PlayerController (이동, 점프, 대시)
세션 2: EnemyAI (순찰, 추적, 공격)
세션 3: UI (체력바, 미니맵, 인벤토리)
세션 4: 사운드 & 이펙트
```

한 세션에서 너무 많은 기능을 다루면 컨텍스트가 발산한다.
`session` 명령으로 끊고 다음 기능으로 넘어가는 것이 효율적.

### 5.2 `read` 명령 활용 패턴

```bash
# ❌ 비효율: 전체 파일 주입
read Assets/Scripts/GameManager.cs

# ✅ 효율: 검색어로 관련 섹션만 주입
read Assets/Scripts/GameManager.cs SpawnEnemy
read Assets/Scripts/PlayerController.cs OnCollision
```

검색어를 주면 150줄 이상 파일에서 관련 부분만 추출한다.

### 5.3 `project update` 타이밍

```
✅ 적절:
  - 새 주요 시스템 추가 (네트워킹, 세이브 시스템 등)
  - 폴더 구조 대규모 변경
  - 프로젝트 목표 변경

❌ 불필요:
  - 기존 스크립트에 메서드 추가
  - 버그 수정
  - UI 미세 조정

project update는 캐시 미스를 유발하므로 아키텍처가 실제로 바뀔 때만 실행.
```

### 5.4 비용 최적화 체크리스트

- [ ] `status` 명령으로 캐시 절약량 확인 (30% 이하면 뭔가 잘못됨)
- [ ] 세션당 요청 10~30개 유지 (너무 길면 압축 빈번 → 맥락 손실)
- [ ] `read`에 검색어 항상 포함
- [ ] `project update`는 아키텍처 변경 시에만
- [ ] 단순 질문(설명, 이유)은 자동으로 800토큰 제한 적용됨

## 6. 파일 구조

```
프로젝트/
├── dev_pipeline_v4.py      진입점, CLI 루프, 세션 관리
├── llm_client.py           API 호출, 캐시 통계, 비용 추적
├── memory_manager.py       5계층 메모리, 압축, 파일 컨텍스트 축약
├── pipeline_controller.py  요청 처리, 상태 생성, diff 적용 트리거
├── file_manager.py         파일 I/O, diff 패치 (백업 포함), 섹션 추출
├── pipeline_config.json    설정 (모델, 경로, 엔진)
├── prompts/
│   └── system.md           시스템 프롬프트 (diff 형식 강제 포함)
└── dev_output/
    ├── PROJECT.md           프로젝트 고정 컨텍스트 (캐시 대상)
    └── SESSION_STATE.md     세션 상태 (세션마다 갱신)
```
